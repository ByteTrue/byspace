# I18n

Client UI translations live in `packages/app/src/i18n`.

## Supported locales

- `ar`
- `en`
- `es`
- `fr`
- `ja`
- `pt-BR`
- `ru`
- `zh-CN`

The persisted language setting is `"system" | SupportedLocale`. `"system"`
checks the browser's locale list in order and falls back to English when none
is supported. Locale resolution is defined in
`packages/app/src/i18n/locales.ts`; update it together with the supported locale
type, selector options, and resources.

Current mappings include language-region variants for Arabic, English,
Spanish, French, Japanese, and Russian; `zh`, `zh-CN`, and `zh-Hans` map to
Simplified Chinese; `pt` and `pt-BR` map to Brazilian Portuguese. Other
Portuguese regional locales remain unsupported.

## What to translate

Translate client-owned copy:

- labels, buttons, headings, and accessibility labels;
- empty, loading, and local status states;
- confirmation text and client-owned error wrappers.

Do not translate runtime or user content:

- agent or daemon output, terminal contents, logs, and code;
- file paths, URLs, commands, provider names, and model names;
- user-authored text or raw protocol/server error text.

Runtime values may be interpolated into a translated client-owned sentence.

## Adding or changing copy

English source strings live in
`packages/app/src/i18n/resources/en.ts`. Every file in
`packages/app/src/i18n/resources/` must have the same key shape and compatible
interpolation placeholders.

Components and hooks use `useTranslation()`. Pure helpers may use the shared
`i18n` instance when they cannot call React hooks. Pass translated labels into
low-level UI primitives unless the primitive owns the copy it renders.

Keep keys grouped by product surface, not component mechanics. When changing a
surface, translate the complete neighboring copy cluster rather than leaving a
mixed-language UI.

## Forge-specific terminology

Indeclinable tokens such as forge brand names, PR/MR initials, and `#`/`!`
number prefixes may be interpolated into one translated key.

Sentences containing the full “pull request” or “merge request” noun use
i18next context. The base key carries pull-request wording and an `_mr` sibling
carries merge-request wording. Callers pass
`getForgePresentation(forge).changeRequestContext` as `context`.

Vocabulary scales by change-request family, not forge. A new forge selects an
existing family in its manifest and does not require duplicate locale keys.

## Validation

Run the focused i18n checks after changing locale policy or resources:

```bash
npx vitest run packages/app/src/i18n/resources.test.ts --bail=1
npx vitest run packages/app/src/i18n/locales.test.ts --bail=1
```

The tests enforce resource parity, interpolation placeholders, and locale
resolution behavior.
